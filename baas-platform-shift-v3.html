01 — Paradigm Shift

The model that
no longer fits.

The traditional bank model is a product-distribution machine — vertically integrated, internally consumed, architecturally closed. What we are building is its inversion: a composable financial platform that any qualified entity can integrate, extend, and build upon.

The model being left behind
✕
Bank as product
Financial services packaged, distributed, and consumed through proprietary channels. The institution is the product.
✕
Monolithic architecture
Systems built to serve internal journeys. Integration with partners is exceptional, slow, and requires custom bilateral work per relationship.
✕
Technology as cost center
IT delivers on business requirements. The platform supports the product — it is not the product. Revenue does not flow through APIs.
✕
Closed financial identity
Customer data, risk models, and compliance infrastructure are locked within the institution. Cannot be composed or extended externally.
✕
Bilateral integration model
Each partner relationship requires dedicated integration work. Does not scale. Every new integration is an N×M complexity problem.
→
The model being built
✓
Bank as platform substrate
Bradesco's balance sheet, risk infrastructure, and regulatory licenses become composable services — accessible via governed APIs to qualified builders.
✓
Layered composable architecture
BaaS + LaaS + CaaS + EaaS as independent, composable layers. Each can be consumed independently or combined — without integration knowledge of the others.
✓
Technology as revenue layer
API consumption generates measurable revenue. Usage-based billing, tiered access, partner analytics. The platform is a product with its own P&L logic.
✓
Portable financial identity
Canonical identity layer (BAAS-ARCH-001) enables a single customer registry shared across all products — portable, governed, composable with zero fragmentation.
✓
Platform integration model
One integration standard, one auth model (OAuth2 + mTLS), one developer contract. Each new partner adds to the ecosystem without adding integration debt.
02 — Architecture

A platform in
four composable layers.

The architectural model is not a monolith being split. It is a new layered system being built alongside the existing institution — designed to be consumed independently, composed freely, and governed centrally.

L0
Real-Time Settlement Infrastructure
Open Cash Hub · IBM Joint Venture · In Production
PIX Cash Management Settlement 100+ Engineers
▾
What it is
Real-time payment infrastructure built as a joint venture with IBM. Handles PIX, cash management, and settlement flows at banking scale. First production installations completed Q2 2026.
Strategic significance
The most operationally complex cluster within Tribo BaaS — regulated, zero-downtime, at the scale of one of Brazil's top-3 banks. It is the financial plumbing that makes everything above possible.
Architecture posture
Distributed atomicity via Saga pattern. PII protection via beneficiary tokenization. Regulatory sequencing: Tributos before dealer repasse per CMN 4.282. Zero tolerance for settlement failure.
L1
Core Platform — BaaS Backbone
Multi-tenant · Auth · Eventing · Observability · Billing
OAuth2 + mTLS Kafka Multi-tenant Squad 1
▾
What it is
The foundational layer all other BaaS products are built on. Canonical identity, auth, multi-tenancy, eventing, observability, and usage metering — enforced uniformly across every product.
Key decisions
BAAS-ARCH-001 (Customer Registry as Canonical Identity) and BAAS-ARCH-002 (Auth & Authorization for Partners). These two architectural records govern the entire platform — not guidelines, enforced constraints.
Stack
Java / Spring Boot. ECS Fargate over EKS. Aurora PostgreSQL Serverless v2. DynamoDB for idempotency via conditional writes. EventBridge over Kafka/MSK. Apigee as MCP Gateway for AI agent access.
L2
Lending as a Service — LaaS
Structured credit · Full origination-to-contract journeys
Capital de Giro Antecipação Crédito Emergencial
▾
What it is
Complete credit journeys — from origination through risk assessment to contract management — exposed as white-label APIs. Partners embed Bradesco's lending capability without owning the risk infrastructure.
Products in scope
Capital de Giro (working capital), Crédito Emergencial, Antecipação de Recebíveis. Each exposed as a composable B2B, B2C, and B2B2B embedded scenario — same underlying platform, different surfaces.
Extensibility model
Credit signal catalog governed by a dedicated squad. New partner segments register new signal codes — no new API versions required. Extensibility is a data problem, not a versioning problem.
L3
Credit as a Service — CaaS
Composable credit primitives · BNPL · Revolving
BNPL Limite Rotativo Composable
▾
What it is
Granular, API-first credit primitives for embedded finance contexts. CaaS ≠ simplified LaaS — it is a distinct architecture designed for non-financial platforms that need to embed credit without full banking integration.
Target contexts
E-commerce (BNPL), subscription platforms (revolving credit), marketplace financing. Any digital context where credit needs to be invisible infrastructure — not a bank product the user visits.
Integration model
White-label MFE components via Module Federation. Design Tokens as runtime CSS Custom Properties — zero rebuild to apply partner brand. Shell knows nothing about the products it hosts.
L4
BaaS + Developer Portal & Open API Surface
API Gateway · Developer Portal · Partner Onboarding · BaaS + Dev Portal Cluster
Axway → Apigee OpenAPI 3.1 DevPortal
▾
What it is
The surface layer where all platform capabilities are exposed — governed, monetized, and consumed by partners. The BaaS + Developer Portal cluster owns this layer, working alongside the Integração Digital cluster to connect Bradesco's internal systems to external partners.
Governance model
Any published API with basepath /mcp automatically becomes an MCP proxy — inheriting auth, quota, and observability policies. AI agents consume the same surface as human developers. One governance layer for everything.
Modernization roadmap
Axway Amplify as current gateway (subject of ongoing modernization proposal). Apigee as the target architecture for unified gateway + MCP proxy + billing. 18-month DX improvement roadmap already benchmarked and sequenced.
03 — Dimensions of Change

Six simultaneous
transformations.

What makes this moment singular is not the scale of a single change — it is the number of dimensions changing simultaneously, and the fact that each one depends on the others to deliver its full value.

DIM 01
Architectural Transformation
From product-siloed systems to a shared platform with composable layers (L0–L4). Multi-tenant isolation, shared identity, standardized auth, centralized observability — patterns that eliminate entire classes of future architectural debt.
DIM 02
Organizational Transformation
A new organizational topology: Tribo BaaS with three independent clusters — Open Cash Hub, Integração Digital, and BaaS + Developer Portal — each with distinct missions, with squads that have product-level ownership and chapters that enforce horizontal standards across the entire Tribe.
DIM 03
Economic Transformation
The move from cost-center IT to a platform P&L model. API consumption becomes measurable, billable, analyzable. Usage-based billing infrastructure is not a future feature — it is being built into Core Platform from day one.
DIM 04
Regulatory Transformation
Open Finance, PIX, CMN 4.282, BACEN requirements — translated into architectural constraints enforced in code, not in audit checklists. Compliance becomes infrastructure. The Aarin relationship executes within a governance model that protects IP and regulatory standing simultaneously.
DIM 05
Partnership Transformation
From bilateral, custom integrations — each a one-off negotiation — to a single integration contract: one auth standard, one API design pattern, one developer experience. Each new partner enriches the ecosystem rather than adding integration debt. The IBM JV (Open Cash Hub) is the first proof of this model at scale.
DIM 06
Intelligence Transformation
GenAI orchestration via Bedrock (generative flows) and classical ML via SageMaker (predictive models, UBI, risk scoring) — separated by design, governed by the same API surface. AI agents consume Bradesco APIs through the same Apigee MCP Gateway as human developers. The platform is AI-native from the architecture up.
