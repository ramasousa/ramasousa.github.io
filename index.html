Developer Portal gRPC

O portal estável atual. Arquivo: dev-portal-v3.html. Estrutura de <article class="article panel"> para todos os painéis — incluindo o Test Clock.

Arquitetura do portal

O portal usa um padrão de SPA (Single-Page Application) com sidebar navigation e painéis que mostram/ocultam via JavaScript. O estado é persistido via Firebase Realtime Database com Anonymous Auth. A função showPanel(id, sbEl, sbId) gerencia a navegação.
Convenção crítica
Painéis devem ser <article class="article panel" id="panel-[name]"> inseridos DENTRO do content-wrap — nunca antes de </body>. Inserção fora do content-wrap quebra o layout de sidebar no mobile.

Decisões técnicas consolidadas

PORTAL-001
Código em <pre> HTML estático — nunca template literals
Backticks não-escapados dentro de template literals JavaScript corrompem silenciosamente todo o script. Blocos de código nos painéis devem ser <pre> estáticos. switchTab() alterna display:none/block, não innerHTML.
PORTAL-002
Botões interativos: sempre <button> — nunca <div onclick>
iOS Safari não dispara eventos de toque em <div> com onclick. Todos os elementos clicáveis (tabs, presets, ações) devem ser <button> com -webkit-appearance:none.
PORTAL-003
Firebase Anonymous Auth com regra auth != null
A regra auth.token.email.matches(/@bradesco\.com\.br/) falha com Anonymous Auth (sem campo email no token). Estado correto e seguro para o protótipo: ".read": "auth != null", ".write": "auth != null". A validação de e-mail corporativo será habilitada com a migração para Entra ID.
PORTAL-004
GitHub Pages CDN (Fastly) tem delay de cache
Arquivos recém-publicados podem aparecer com conteúdo antigo. Usar nomes com versão (dev-portal-v3.html, não index.html) e verificar via URL direta após Actions check verde.
Sandbox · Simulação Temporal

Test Clock API

Painel nativo dentro do portal gRPC — não uma página separada. Permite avançar o relógio do sandbox sem esperar tempo real, simulando vencimentos, cobranças e expiração de tokens.
Estado atual
O painel existe como mockup de design e especificação viva. Os endpoints /sandbox/v1/clock não existem no backend — são o contrato que orienta a implementação. A squad A4 (DX & Confiabilidade) implementa o backend estimando 6-10 semanas, com a principal complexidade sendo mapear todos os now() no código do sandbox.

Padrão Stripe — o que é o Test Clock

Originado na Stripe, o Test Clock permite que o sandbox avance no tempo de forma controlada. O parceiro cria uma proposta de crédito com vencimento em 30 dias, avança o relógio +30 dias, e observa o webhook de cobrança sendo disparado — sem esperar o mês real passar.

Casos de uso validados

💳 Capital de Giro
Vencimento em 30 dias → avança +30d → valida webhook de cobrança.
⏰ Expiração de Proposta
Proposta expira em 7 dias → avança +7d → valida comportamento de não-assinatura.
🔑 Token OAuth
Refresh token expira em 90 dias → avança +90d → valida fluxo de renovação.

Partner Success · Operacional

Playbook Operacional

A tecnologia identifica o parceiro em risco. O processo humano age. Sem esse elo, o dado vira relatório que ninguém lê.

Os 3 gatilhos

🏖️ Sandbox Stall
Threshold: ≥7 dias sem evolução. Playbook de 5 passos (Dia 7 e-mail auto → Dia 10 pessoal → Dia 14 ligação → Dia 18 escalada → Dia 21 decisão). SLA: 7 dias.
🚨 Degradação em PRD
Threshold: error_rate >5% por 15min. Playbook de 4 passos (t=0 webhook auto → t=15min diagnóstico → t=30min ação → t=2h status update). SLA: 30 min.
📉 Volume Drop
Threshold: volume <70% da média por 3 dias. Playbook: Dia 3 flag interno → Dia 5 contato consultivo → Dia 10 comercial. SLA: 5 dias.

Segurança Firebase — estado atual

Regra correta (aplicada)
{".read":"auth != null",".write":"auth != null"} — exige sessão autenticada. Melhor que aberto. A validação de e-mail @bradesco.com.br só funciona com Google Auth ou Email/Password Auth, não com Anonymous Auth (token não carrega campo email).

Sequência de migração recomendada

Hoje
Firebase + auth != null
Seguro suficiente para protótipo interno. Chave restrita ao domínio ramasousa.github.io no Google Cloud Console.
Próximo
Hospedar internamente
Azure Static Web Apps em domínio *.bradesco.com.br. HTML não muda — apenas a origem muda.
Médio prazo
Spring Boot + Cosmos DB + Entra ID
Substitui Firebase de uma vez. HTML vira thin client. Entra ID já existe no Bradesco — não há novo sistema de identidade a configurar.
Princípio inegociável
O HTML deve ser um thin client que chama uma API REST. Nunca acessar o banco de dados diretamente do frontend. É o mesmo padrão que o BaaS usa para parceiros externos — e é o que permite auditoria, controle de acesso e escalabilidade.

ADRs · Architecture Decision Records

Decisões técnicas

Decisões consolidadas durante a construção. Algumas são atualizações das ADRs originais da v1.0, outras são novas.

ADRs originais (v1.0) — mantidas

ADR-001
URL Pattern: /v{n}/produto/recurso/{id}/ação
Versionamento major na URL. Minor/patch via changelog sem mudança de URL.
ADR-002
Erros: RFC 7807 com campo code em SNAKE_CASE
Campo code é string estável para lógica de negócio. Inclui traceId para correlação de logs.
ADR-003
IdempotencyKey obrigatório em todas as escritas
GUID v4 único por requisição POST/PUT. Validade 24h. Reutilização retorna resultado original.
ADR-005
Design-First: spec OpenAPI 3.1 antes da página de docs
A spec é a fonte de verdade. Docs, exemplos e Try-it são derivados da spec.

ADRs novas (v2.0)

ADR-007
HTML é thin client — nunca acessa banco diretamente
Toda lógica de dados passa por uma API REST Spring Boot. O HTML não tem credenciais de banco de dados. Aplica-se a Firebase (interim) e Cosmos DB (produção).
ADR-008
Código em <pre> estático — nunca template literals
Backticks não-escapados dentro de template literals JavaScript corrompem silenciosamente o script inteiro. Todo código exibido ao usuário deve estar em <pre> HTML. Tabs alternam com display:none/block, não com innerHTML = CODE[lang].
ADR-009
Botões interativos: sempre <button>
iOS Safari não dispara eventos de toque em <div onclick>. Todos os elementos clicáveis usam <button> com -webkit-appearance:none no CSS.
ADR-010
Extensibilidade via catálogo — não versão de API
Novos segmentos de parceiros registram novos signal codes no catálogo de crédito — não criam uma nova versão da API. Mantém a API estável enquanto o catálogo cresce.
ADR-011
Aarin: Modelo Híbrido com Governança
Recomendação consolidada: Aarin executa, Bradesco retém IP e governa. Não delegar o roadmap do produto ao parceiro operacional.
ADR-012
Firebase Anonymous Auth com auth != null (interim)
A regra com email.matches(/.*@bradesco\.com\.br/) não funciona com Anonymous Auth — o token não carrega campo email. Estado correto: auth != null até migração para Entra ID.

// Node.js (axios) — obter Bearer Token
const axios = require('axios');
const qs = require('querystring');

async function getToken() {
  const { data } = await axios.post(
    'https://auth.bradesco.com.br/oauth2/token',
    qs.stringify({
      grant_type: 'client_credentials',
      client_id: process.env.BRADESCO_CLIENT_ID,
      client_secret: process.env.BRADESCO_CLIENT_SECRET,
      scope: 'caas:read caas:write payments:read payments:write'
    }),
    { headers: { 'Content-Type': 'application/x-www-form-urlencoded' } }
  );
  return data.access_token; // expira em 3600s
}

// Java (Spring RestTemplate) — obter Bearer Token
@Service
public class BradescoAuthService {
    @Value("${bradesco.client-id}")  private String clientId;
    @Value("${bradesco.client-secret}") private String clientSecret;

    public String getToken() {
        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("grant_type", "client_credentials");
        body.add("client_id", clientId);
        body.add("client_secret", clientSecret);
        body.add("scope", "caas:read caas:write payments:read payments:write");

        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);

        ResponseEntity<Map> response = restTemplate.exchange(
            "https://auth.bradesco.com.br/oauth2/token",
            HttpMethod.POST, new HttpEntity<>(body, headers), Map.class);
        return (String) response.getBody().get("access_token");
    }
}

Webhook — verificar assinatura HMAC

// Node.js (Express) — Webhook com HMAC-SHA256
const crypto = require('crypto');

app.post('/webhook/bradesco', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-bradesco-signature'];
  const secret    = process.env.WEBHOOK_SECRET;
  
  // timing-safe comparison — NUNCA usar === para assinaturas
  const expected = crypto
    .createHmac('sha256', secret)
    .update(req.body)
    .digest('hex');
  
  const valid = crypto.timingSafeEqual(
    Buffer.from(signature, 'hex'),
    Buffer.from(expected,  'hex')
  );
  
  if (!valid) return res.status(401).json({ error: 'Assinatura inválida' });
  
  const event = JSON.parse(req.body);
  console.log(`Evento: ${event.type} — ${event.resourceId}`);
  res.status(200).json({ received: true });
});
# Python (FastAPI) — Webhook com HMAC-SHA256
import hmac, hashlib, os
from fastapi import Request, HTTPException

@app.post("/webhook/bradesco")
async def webhook(request: Request):
    body      = await request.body()
    signature = request.headers.get("x-bradesco-signature", "")
    secret    = os.environ["WEBHOOK_SECRET"].encode()

    expected = hmac.new(secret, body, hashlib.sha256).hexdigest()

    # compare_digest previne timing attacks
    if not hmac.compare_digest(signature, expected):
        raise HTTPException(status_code=401, detail="Assinatura inválida")

    event = await request.json()
    print(f"Evento: {event['type']} — {event['resourceId']}")
    return {"received": True}
// Java (Spring Boot) — Webhook com HMAC-SHA256
@PostMapping("/webhook/bradesco")
public ResponseEntity<?> webhook(
    @RequestHeader("X-Bradesco-Signature") String signature,
    @RequestBody byte[] body) throws Exception {

    String secret = System.getenv("WEBHOOK_SECRET");
    Mac mac = Mac.getInstance("HmacSHA256");
    mac.init(new SecretKeySpec(secret.getBytes(), "HmacSHA256"));
    String expected = Hex.encodeHexString(mac.doFinal(body));

    // MessageDigest.isEqual previne timing attacks
    if (!MessageDigest.isEqual(expected.getBytes(), signature.getBytes()))
        return ResponseEntity.status(401).body("Assinatura inválida");

    Map<?,?> event = objectMapper.readValue(body, Map.class);
    log.info("Evento: {} — {}", event.get("type"), event.get("resourceId"));
    return ResponseEntity.ok(Map.of("received", true));
}

Migração para ReadMe.io

O ReadMe.io é o padrão da indústria para developer portals — Stripe, Twilio e Plaid usam. A migração do HTML atual é direta e pode ser feita em 2-3 dias.

Por que migrar?

O que você ganha
Busca semântica nativa (AI-powered), Try-it automático via spec OpenAPI, changelog gerado por release, versionamento de docs, analytics de uso por endpoint, comunidade de devs integrada.
O que você perde (temporariamente)
Controle total do CSS e do layout. O Design System Bradesco precisará ser adaptado para o sistema de customização do ReadMe (cores, fontes, logo). Sem o Antes × Depois nativo.

Passo a passo da migração

Passo 1
Instalar o rdme CLI
npm install -g rdme — ferramenta oficial do ReadMe para sincronizar specs e docs.
Passo 2
Subir os specs OpenAPI
rdme openapi specs/caas/openapi.yaml --key=<API_KEY> --id=<API_ID> — repita para cada produto (CaaS, PIX, Auth, LaaS).
Passo 3
Sincronizar os docs Markdown
rdme docs docs/caas/ --version=v1.0 --key=<API_KEY> — o Markdown existente vai direto. Links entre páginas precisam ser ajustados.
Passo 4
Customizar o Design System
ReadMe → Appearance → Custom CSS. Aplicar as variáveis de cor do Bradesco (--primary: #CC092F) e a fonte Bradesco Sans (Inter como fallback).
Passo 5
Configurar CI/CD
Adicionar o step rdme no GitHub Actions (ver seção CI/CD). A partir daí, push em main sincroniza docs e specs automaticamente.

Console · Developer Experience

Console — propósito e gestão

O Console do Developer Portal é onde o parceiro gerencia suas credenciais, monitora uso e acessa o sandbox. É diferente do Console Avançado (interno) — este é o console voltado ao parceiro.

Console do parceiro vs Console Avançado

Console do Parceiro (Developer Portal)
Gerenciar credenciais (Client ID/Secret), ver uso de APIs próprias, acessar sandbox, gerenciar webhooks e ver histórico de chamadas. Público-alvo: o desenvolvedor do parceiro.
Console Avançado (Interno)
Visão cross-partner: funil de onboarding, alertas de degradação, comparação de latência, heatmap de endpoints. Público-alvo: equipe interna (ops, Partner Success). Ver seção Console Avançado.

Como o console do parceiro é gerenciado

No modelo atual com Axway Amplify, o console do parceiro é o Amplify Engage — o portal de marketplace nativo da plataforma. O parceiro cria uma App, subscreve um API Product e recebe as credenciais automaticamente. A squad DX Platform configura os Products e Plans no Amplify; a squad A4 mantém o portal customizado que orquestra essa experiência.

Como gerar um SDK a partir da spec

Shell
# Gerar SDK Node.js para CaaS
npx @openapitools/openapi-generator-cli generate \
  -i specs/caas/openapi.yaml \
  -g typescript-axios \
  -o sdks/caas-node \
  --additional-properties=npmName=@bradesco/caas-sdk,npmVersion=1.0.0

# Gerar SDK Python para CaaS
openapi-generator-cli generate \
  -i specs/caas/openapi.yaml \
  -g python \
  -o sdks/caas-python \
  --additional-properties=packageName=bradesco_caas

# Testar o SDK gerado
cd sdks/caas-node && npm install && npm test
Ferramentas · Testabilidade

Postman Hub

O Postman Hub centraliza todas as Collections da plataforma Bradesco — uma por produto, com variáveis de ambiente pré-configuradas para sandbox e produção.

Estrutura das Collections

Cada produto tem uma Collection Postman organizada espelhando a estrutura da documentação: Autenticação → Simulação → Proposta → Garantias → Assinatura → Webhooks. As variáveis de ambiente (baseUrl, clientId, clientSecret, token) são compartilhadas entre todos os produtos.

Como exportar a Collection da spec

Shell · Postman CLI
# Instalar Postman CLI
npm install -g @postman/cli

# Importar spec OpenAPI como Collection
postman collection from-file specs/caas/openapi.yaml \
  --output collections/caas.json

# Publicar no Postman Hub (requer API Key)
postman collection push collections/caas.json \
  --api-key $POSTMAN_API_KEY \
  --workspace "Bradesco Developer Portal"

# Rodar testes automatizados
postman collection run collections/caas.json \
  --environment environments/sandbox.json

IA · Developer Experience

Chatbot — conectar com IA

Um chatbot de documentação reduz o tempo de resposta para perguntas técnicas de horas (suporte humano) para segundos. A abordagem é RAG (Retrieval-Augmented Generation) com as specs e docs do portal como fonte de verdade.
Bradesco já lidera em AI-native (MCP)
O DX Score benchmark mostra que o Bradesco já supera o Stripe na dimensão AI-native graças ao MCP Server nativo integrado ao portal. O chatbot de documentação é o próximo passo natural — e uma vantagem competitiva real frente ao Itaú e outros pares bancários.

Arquitetura recomendada

Opção A · Mais rápida
ReadMe.io AI Search (nativo)
Ao migrar para ReadMe.io, a busca semântica AI vem incluída. Indexa specs, docs e changelogs automaticamente. Tempo para ativar: 1 configuração. Sem custo adicional nos planos Enterprise.
Opção B · Mais controle
Claude API + RAG sobre specs OpenAPI
Pipeline: specs OpenAPI 3.1 → embeddings (ex: text-embedding-3-small) → vetor store (Pinecone ou pgvector) → Claude API como LLM. Permite customizar o comportamento, restringir ao escopo da plataforma e adicionar contexto de sandbox.
Opção C · Já implementado
MCP Server do portal gRPC
O portal gRPC já tem um MCP Server que expõe as ferramentas da plataforma para clientes MCP (Claude Desktop, Cursor, etc.). Parceiros técnicos podem usar via protocolo MCP — sem chatbot UI, mas com capacidade equivalente para IDEs.
